FAQ

The questions senior engineers ask.

Won't it go off the rails and invent bad architecture?

You own the architecture at design-time. gitban generates and you approve PRDs, ADRs, and design docs before any code is written — agents execute a vetted plan, not a vibe.

How do I trust / review what it did?

The reasoning ships with the code. Every structural decision is checked into the repo as plain text beside the diff — a human-readable audit trail that "reads like the mind of an engineer." No black box.

Parallel agents — isn't that merge-conflict and race-condition hell?

Worktree isolation is the design, not a side effect. Each agent works on its own clean, isolated branch and is destroyed when its task passes the gate. The context is clean every time.

Will it actually be production-grade?

Quality is the output of the process, not a cleanup afterward: TDD at design-time, adversarial review loops, and validation gates that block "done" until every check passes.

Will my code leave my machine?

No. gitban is hermetic and runs entirely offline with zero telemetry. Your code and IP never leave your machine — there is no cloud to leak to.

Won't token costs explode?

The opposite. Disposable, scoped agents get only the context they need instead of brute-forcing a giant window — so spend converts into shipped output instead of a runaway bill.

When do I have to pay?

Only when gitban is used by or within an organization for commercial work. Free for any individual, and free for non-commercial / open-source work at any headcount. A commercial-use license, not a feature wall.

Is it open source? Does it phone home?

No to both. gitban ships as a closed binary (source available to Enterprise under NDA for audit), and runs entirely offline with no network calls.
